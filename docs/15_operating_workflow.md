# 百果山探索樂園顧問案作業流程

> 狀態標籤：已確認

## 文件目的

這份文件用來固定百果山探索樂園顧問案後續的作業方式。之後每次開會、補資料、更新報告，都比照這份流程執行，並以目前這個專案作為唯一更新基準。

## 目前建議做法

現階段建議採用：

**專案內 SOP 文件 + 固定資料夾結構**

而不是先做成 skill 或自動化 workflow。

## 為什麼現在先不要急著做成 skill

原因有三個：

1. 這個顧問案目前還在持續修流程
2. 報告格式、資料欄位、業主需求都還可能調整
3. 太早封裝成 skill，後面每改一次流程就要同步改工具

## 後續最穩的執行方式

### 1. 以這個專案為主體

所有後續顧問資料與報告，都以這個專案為主：

- `docs/`：知識庫與規則
- `meeting-reports/`：每次會議報告
- `monthly-reports/`：每月報告
- `quarterly-reports/`：每季顧問報告
- `client-report/`：給業主看的網頁版報告
- `content-plans/`：給內部小編與顧問端對齊的內容規劃

### 1.1 內部協作頁面

除業主版頁面外，後續會再建立：

- `war-room/`：99yi顧問小編 戰情室

用途不是給業主看，而是讓內部社群小編、顧問端與專案負責人可以查看：

- 專案現況
- 業主最新資訊
- 會議紀錄
- 最新社群規劃
- 可設計議題
- 尚未補齊的資料

目前 MVP 已建置完成，可作為內部小編與顧問端的固定工作頁面。

### 2. 每次開會前固定流程

1. 先確認本機日期與時間
2. 確認本次報告期間與資料截點
3. 先更新已確認的公開資料
4. 整理業主新提供的內部數據
5. 更新會議報告與對應的月報 / 季報
6. 同步更新業主版頁面

### 3. 每次開會時固定做的事

1. 先報告目前已整理完成的內容
2. 明確指出哪些數據已確認
3. 明確指出哪些資料仍需業主提供
4. 確認下次更新前要補哪些數據
5. 確認下一次報告要追的重點與專題

### 4. 每次開會後固定流程

1. 補會議摘要
2. 整理業主新增資料
3. 更新月報 / 季報
4. 更新業主版頁面
5. 保留時間標記、資料截點與版本號

## 固定生成的文件

### 每次會議

- 會議摘要
- 若有重大更新，同步更新業主版頁面

### 每月

- 月報

### 每季

- 季報
- 本季專題議題
- 業主版季度頁面更新

## 每次更新都要遵守的規則

1. 先確認本機時間
2. 所有報告都要有時間標記
3. 所有重要數據都要標示：
   - 已確認
   - 推估
   - 待驗證
4. 沒查核過的數字不得直接寫成正式結論
5. 每季除了固定 KPI 外，還要有一個新的專題議題

## 什麼時候再做成 skill 或 workflow

等以下條件成立，再考慮把它封裝成 skill 或 workflow：

1. 連續執行 2 到 3 個月都沒有再大改流程
2. KPI 欄位固定
3. 業主提供資料的格式固定
4. 會議摘要、月報、季報版型固定

## 未來可封裝的方向

如果之後流程穩定，可以再做兩種工具化：

### A. skill

用途：

- 快速依固定規則生成會議摘要、月報、季報
- 自動提醒要檢查時間、資料截點、版本號

### B. workflow

用途：

- 固定建立新一期報告檔案
- 自動套入模板
- 自動更新業主版頁面中的季度切換與內容區塊

## 當前結論

目前最好的做法不是先做工具，而是先把流程寫死在專案內，並且之後每次都照這份 SOP 執行。

換句話說：

**後續開會就直接比照這份流程，用這個專案持續更新。**
