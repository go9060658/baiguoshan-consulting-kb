# 99yi顧問小編 戰情室｜頁面規劃

> 狀態標籤：已確認、已建置 MVP

## 一、頁面定位

這個頁面不是給業主看的季度報告，而是給 **99yi 內部社群小編與顧問端** 共同使用的工作頁。

它的任務有三個：

1. 把每次跟業主開會後的重要資訊同步給小編
2. 把最新的社群規劃、可執行題型與注意事項集中在同一頁
3. 讓不同專案都能用同一套結構查看，不用每次靠訊息轉述

頁面名稱固定為：

**99yi顧問小編 戰情室**

## 一之一、與業主頁的分流原則

戰情室與業主頁必須明確分開。

### 原則

1. 業主頁不能有任何連結導向戰情室
2. 戰情室可以連到業主頁作為參考，但反向不可導流
3. 戰情室不跟目前公開的 GitHub Pages 業主主站一起發布
4. 若未來要提供內部團隊外部存取，需用另一個受控部署位置

### 目前做法

- 業主頁：維持公開 GitHub Pages
- 戰情室：保留於專案內，作為內部頁面使用

## 二、這個頁面要解決的問題

目前你和業主開完會之後，資訊會先進專案，再由你口頭或訊息轉給公司內部小編。這種做法有三個問題：

1. 資訊容易斷層
2. 小編不一定知道哪一份是最新版本
3. 顧問策略、會議結論與內容規劃會分散在不同文件

所以這個戰情室要做的不是「再做一份報告」，而是把內部執行最需要看的東西做成固定頁面。

## 三、主要使用者

### 1. 你自己

用途：

- 開會後更新資料
- 查看目前每個專案的最新狀態
- 確認下一步要交代給小編什麼

### 2. 社群小編

用途：

- 快速看懂目前專案最新方向
- 知道這週該做什麼內容
- 知道哪些資訊已確認、哪些還不能亂寫

### 3. 顧問協作成員

用途：

- 查會議結論
- 查目前業主狀況
- 查最新策略與建議題型

## 四、頁面層級設計

這個頁面建議做成「一個總入口 + 專案切換 + 分頁內容」的形式。

## 五、首頁要有的固定元素

### A. 頁面標題區

固定顯示：

- 99yi顧問小編 戰情室
- 最後更新時間
- 本次資料截點
- 當前版本號

### B. 專案切換區

這裡是整個頁面的核心，必須可以切換不同專案。

建議形式：

- 專案下拉選單
- 或卡片式專案列表

每個專案卡至少要有：

- 專案名稱
- 專案狀態（進行中 / 待啟動 / 暫停）
- 最新更新時間
- 本期主題
- 下一步動作

### C. 分頁切換區

同一個專案切進去後，內容要用分頁整理，不要做成一整條長頁面。

## 六、建議分頁架構

以下是建議的固定分頁。

### 1. 專案總覽

用途：

- 給小編快速看懂這個專案現在在做什麼

要放的內容：

- 專案一句話定位
- 最新顧問判斷
- 本期重點主軸
- 目前最重要的三件事
- 本週 / 本月優先任務

### 2. 業主資訊

用途：

- 集中放小編需要知道的業主相關背景與目前確認事項

要放的內容：

- 業主目前經營重點
- 已確認數據摘要
- 目前已確認規劃
- 業主明確不想做的事
- 需注意的溝通語氣

### 3. 會議紀錄

用途：

- 讓內部成員直接查每次和業主開會後的結論

要放的內容：

- 每次會議列表
- 會議日期
- 會議重點摘要
- 本次新增資訊
- 本次決策
- 會後待辦

建議功能：

- 只顯示最新三次
- 可點開看完整紀錄

### 4. 最新社群規劃

用途：

- 給小編直接看目前最新的內容方向

要放的內容：

- 本季社群目標
- 本月內容主軸
- 貼文比例
- 本週應優先產出類型
- 應避免的方向
- 特殊節日與檔期規劃

### 5. 可設計議題庫

這個分頁很重要，因為小編最需要的是「現在可以做什麼題」。

要放的內容：

- 已確認可做題型
- 待發展題型
- 節日題型
- 安全教育題型
- 互動型題型
- 夜間營運題型

每個題型卡建議至少要有：

- 題型名稱
- 類別
- 用途
- 參考文案方向
- 建議形式（圖文 / Reels / 限動 / 短片）
- 狀態（可直接做 / 需等業主確認 / 暫不建議）

### 6. 後續需補資料

用途：

- 讓內部知道哪些資料還缺
- 也方便你開會後直接整理給業主追件

要放的內容：

- 人流資料
- 營收資料
- 社群後台資料
- 現場回饋
- 特定活動成效資料

每一項應標示：

- 是否已收到
- 最新更新日期
- 缺什麼

### 7. 更新日誌

用途：

- 讓內部成員知道這一版和上一版差在哪裡

要放的內容：

- 更新日期
- 更新人
- 更新內容摘要
- 影響哪些分頁

## 七、每個專案都要有的固定欄位

為了讓未來不同專案都能共用同一套結構，建議每個專案都固定有以下欄位：

- 專案名稱
- 專案代號 / slug
- 業主名稱
- 專案狀態
- 報告基準時間
- 最新會議日期
- 最新更新時間
- 最新顧問判斷
- 本期內容主軸
- 本期需補資料

## 八、頁面視覺建議

這頁是內部作戰頁，不需要做成像提案簡報那樣華麗，但也不能只是文件列表。

建議視覺方向：

- 看起來像內部戰情面板
- 卡片式區塊
- 清楚的狀態標籤
- 專案切換很明顯
- 一眼看出最新更新時間

### 視覺元素建議

- 專案狀態 badge
- 已確認 / 待確認 / 待執行 標籤
- 重要提醒區塊
- 本週重點區塊
- 待補資料警示區塊

## 九、資料更新流程

這個頁面未來的更新流程，應和你現在的顧問專案 SOP 接上。

### 每次和業主開會後

1. 先確認本機時間
2. 先更新會議紀錄
3. 更新 Q2 / 月報 / 季報等正式文件
4. 抽出對小編有用的內容，更新到戰情室
5. 更新「後續需補資料」
6. 更新「可設計議題庫」

### 每次社群策略有新想法時

1. 先判斷屬於哪個專案
2. 寫入該專案的題型庫或最新社群規劃
3. 標示是：
   - 已確認可做
   - 待業主確認
   - 僅顧問構想

## 十、建議的檔案結構

如果之後真的開始做這個頁面，建議在專案內新增一個獨立資料夾：

```text
war-room/
  index.html
  styles.css
  app.js
  data/
    projects/
      baiguoshan.json
      other-project.json
```

### 為什麼要獨立出來

因為這頁面是給內部使用，和業主版頁面不同，不應混在 `client-report/` 裡。

## 十一、MVP 第一版一定要先做的功能

第一版不需要太複雜，但這些功能一定要有：

1. 專案切換
2. 分頁切換
3. 最新更新時間
4. 會議紀錄列表
5. 最新社群規劃
6. 可設計議題卡片
7. 後續需補資料清單

## 十二、第二階段可以再加的功能

流程跑順之後，可以再補：

1. 關鍵字搜尋
2. 按狀態篩選題型
3. 本週待辦區塊
4. 依平台分類（FB / IG / Reels / Shorts）
5. 每月內容行事曆
6. 可直接複製的小編文案草稿

## 十三、顧問建議

這個頁面最重要的不是漂亮，而是：

**讓你和小編對同一個專案，永遠看的是同一版最新資訊。**

所以第一版應優先確保三件事：

1. 結構清楚
2. 更新方便
3. 看得出什麼是最新、什麼是已確認、什麼還不能做

## 十四、目前建議結論

這個頁面目前已完成 MVP 第一版，後續可以直接沿用同一套方式擴充其他專案。

目前的正確使用方式是：

1. 開會後先更新專案內正式文件
2. 再把對小編有用的重點同步到戰情室
3. 後續新增其他專案時，沿用同一套專案資料結構與分頁邏輯

換句話說：

**99yi顧問小編 戰情室 現在已經是一個可用的「內部多專案作戰頁 MVP」。**
